# Convention Pull Requests

## Status

Draft.

## Purpose

`repo-conventions` already supports creating or updating a pull request with `apply --open-pr`.

This document proposes a unified first version of pull request behavior for generated conventions PRs. It consolidates two related concerns under one `pull-request` configuration surface:

- pull request metadata such as labels, reviewers, and assignees
- auto-merge behavior and merge-method selection

The goal is to keep all PR-facing policy in one place while preserving a clear distinction between additive metadata and the separate decision about whether the PR should auto-merge.

## Goals

- Always add a stable `repo-conventions` label to generated conventions PRs.
- Allow repositories to configure additional labels, reviewers, and assignees for generated conventions PRs.
- Allow individual convention references to contribute pull request metadata and auto-merge preferences.
- Ensure convention-level pull request behavior is only contributed by conventions that actually produce commits in the generated PR.
- Allow repository operators to opt into auto-merging conventions PRs.
- Allow callers to override auto-merge behavior for a specific run.
- Support GitHub's merge methods: merge, squash, and rebase.
- Keep pull request creation or update resilient when the requested merge method is not allowed by the repository.
- Avoid replacing the existing title and branch naming conventions as the primary PR identity.

## Non-Goals

- Bypassing repository branch protection or required checks.
- Auto-approving pull requests.
- Inferring labels, reviewers, or assignees from changed files or convention names.
- Removing unrelated human-added labels, reviewers, or assignees from an existing PR.
- Per-file or per-change safety heuristics beyond whether a convention actually produced commits.
- Supporting different auto-merge settings for different conventions within the same generated PR.

## Background

Today, conventions PRs are already identifiable because they use:

- the stable title `Apply repository conventions.`
- the `repo-conventions` branch family
- a PR body that states the PR was generated by `repo-conventions`

That means labels and other metadata improve triage and automation, but they do not replace the current identity model.

The missing capability is a unified policy model. Repositories need one place to say:

- what metadata should appear on generated PRs
- whether generated PRs should auto-merge
- which merge method should be preferred

## Proposed CLI Surface

Extend the `apply` command with these options:

- `--open-pr`: Existing behavior. Creates or updates the conventions pull request.
- `--auto-merge`: After creating or updating the pull request, request GitHub auto-merge.
- `--no-auto-merge`: Explicitly disable auto-merge for this run, even if configuration would otherwise enable it.
- `--merge-method <merge|squash|rebase>`: Choose the preferred merge method for this run.

Rules:

- `--auto-merge` and `--no-auto-merge` are mutually exclusive.
- `--auto-merge` implies `--open-pr`.
- `--merge-method` does not by itself imply auto-merge, but it influences method selection when auto-merge is enabled by either CLI or configuration.
- `--no-auto-merge` does not imply `--open-pr`; it only affects runs that already create or update a PR.

## Proposed Configuration Surface

Add a top-level `pull-request` section in `.github/conventions.yml`.

### Repository Configuration

```yaml
pull-request:
  labels:
    - automation
  reviewers:
    - octocat
    - my-org/build-team
  assignees:
    - octocat
  auto-merge: true
  merge-method: squash

conventions:
  - path: owner/shared-conventions/dotnet-format@v1
```

Proposed semantics:

- `pull-request.labels` adds repository-level labels to generated conventions PRs.
- `pull-request.reviewers` adds repository-level reviewers to generated conventions PRs.
- `pull-request.assignees` adds repository-level assignees to generated conventions PRs.
- `pull-request.auto-merge` sets the repository default for generated conventions PRs.
- `pull-request.merge-method` sets the repository's preferred merge method when auto-merge is enabled.
- If omitted, the corresponding setting contributes nothing.

### Convention Configuration

Allow each convention reference to contribute pull request behavior:

```yaml
conventions:
  - path: owner/shared-conventions/dotnet-format@v1
    pull-request:
      labels:
        - dependencies
      reviewers:
        - my-org/dotnet-team
      assignees:
        - octocat
      auto-merge: true
      merge-method: squash
```

Proposed semantics:

- Convention-level pull request settings are attached to the convention reference in the consuming repository.
- Convention-level pull request settings only contribute when that convention produces one or more commits in the generated PR.
- Convention-level metadata is additive with repository-level metadata.
- Convention-level auto-merge settings contribute to the single effective auto-merge decision for the PR.
- The consuming repository remains in control; remote convention repositories do not directly impose PR behavior.

## Contributing Conventions

This document uses the term `contributing convention` to mean a convention reference that produced one or more commits in the generated PR.

Rules:

- Repository-level pull request settings always apply.
- Convention-level pull request settings are ignored for conventions that ran but produced no commits.
- If no conventions produced commits, no PR is created or updated, so pull request behavior does not apply.

This keeps convention-level PR behavior aligned with the actual changes under review.

## Effective Metadata Model

For labels, reviewers, and assignees, the right reduction model is additive union.

### Labels

The effective label set for a generated PR is:

1. The implicit built-in label `repo-conventions`.
2. Repository-level `pull-request.labels`.
3. Convention-level `pull-request.labels` from contributing conventions.

Rules:

- Labels are deduplicated case-insensitively.
- Existing label casing is preserved when a matching repository label already exists.
- The built-in `repo-conventions` label is always included even if it is not present in configuration.

### Reviewers

The effective reviewer set is the union of:

1. Repository-level `pull-request.reviewers`.
2. Convention-level `pull-request.reviewers` from contributing conventions.

Rules:

- Reviewers are deduplicated.
- Reviewer values may represent GitHub users or review-requestable teams.
- The CLI adds configured reviewers but does not remove existing reviewers.

### Assignees

The effective assignee set is the union of:

1. Repository-level `pull-request.assignees`.
2. Convention-level `pull-request.assignees` from contributing conventions.

Rules:

- Assignees are deduplicated.
- Assignee values are GitHub users.
- The CLI adds configured assignees but does not remove existing assignees.

## Built-In Label

The built-in `repo-conventions` label behaves specially.

If it does not already exist in the repository, `repo-conventions` should create it with:

- Color: `1d76db`
- Description: `Pull requests generated by repo-conventions.`

If the label already exists:

- Reuse it.
- Do not change its color.
- Do not change its description.

## Label Creation And Identity Validation

Before creating or updating the PR:

- Resolve the effective label set.
- Create any missing labels.
- For user-specified labels, use GitHub's default color and default empty description.
- For the built-in `repo-conventions` label, use the standard color and description above.
- Never overwrite an existing label's color or description.

Reviewers and assignees cannot be auto-created, so they need validation instead.

Proposed behavior:

- Validate reviewer and assignee identities before mutating the PR when GitHub exposes that information cheaply.
- If a configured reviewer or assignee is invalid or cannot be requested for the target repository, fail with a clear message.
- Do not silently skip invalid reviewers or assignees.

## Effective Auto-Merge Model

The CLI computes one effective auto-merge decision for the entire PR.

### Auto-Merge Enablement

Precedence, highest to lowest:

1. CLI override.
2. Repository-level `pull-request.auto-merge`.
3. Explicit convention-level `pull-request.auto-merge` from contributing conventions.
4. Convention-type defaults inferred from contributing conventions.
5. Default behavior: auto-merge disabled.

Convention-type defaults:

- Executable convention: default to `false` for safety.
- Composite convention: default to `true` only if all of its contributing child conventions resolve to `true`.

Reduction rules when repository policy does not decide the outcome:

- If any contributing convention explicitly sets `pull-request.auto-merge: false`, the effective value becomes `false`.
- Otherwise, if one or more contributing conventions explicitly sets `pull-request.auto-merge: true`, the effective value becomes `true`.
- Otherwise, if every contributing convention resolves to a convention-type default of `true`, the effective value becomes `true`.
- Otherwise, the effective value remains `false`.

This keeps `false` as the safer conflict resolver.

### Merge Method Selection

First compute the desired merge method using this precedence order:

1. CLI `--merge-method`.
2. Repository-level `pull-request.merge-method`.
3. The single explicit convention-level `pull-request.merge-method` if all contributing conventions that specify one agree.
4. Default desired method: `squash`.

If contributing conventions disagree on merge method and repository policy does not settle the conflict, the desired method becomes `squash` and the CLI should print a note.

### Merge Method Fallback

If the desired merge method is not allowed by the repository, the CLI must not fail PR creation or update because of that rejection.

Fallback order:

1. Try the desired method.
2. If that method cannot be used, try `squash`.
3. If `squash` cannot be used, try `merge`.
4. If `merge` cannot be used, try `rebase`.

Rules:

- Do not try the same method twice.
- If the desired method is already `squash`, the next fallback is `merge`, then `rebase`.
- If the desired method is already `merge`, the next fallback is `squash`, then `rebase`.
- If the desired method is already `rebase`, the next fallback is `squash`, then `merge`.

Preferred behavior:

- Detect allowed merge methods up front when GitHub exposes that capability cheaply.
- If that information is unavailable, stale, or incomplete, attempt the desired method and reactively retry using the fallback order above.

This hybrid approach keeps the happy path predictable while ensuring that an unsupported method does not block PR creation.

## GitHub Integration Behavior

When `apply --open-pr` creates a new PR:

- Compute the effective metadata and effective auto-merge behavior.
- Create any missing labels.
- Validate reviewers and assignees.
- Create the PR with the effective labels, reviewers, and assignees.
- If auto-merge is enabled, attempt to enable it after the PR exists.

When `apply --open-pr` updates an existing PR:

- Recompute the effective metadata and effective auto-merge behavior.
- Create any newly needed labels.
- Validate reviewers and assignees.
- Add any missing labels, reviewers, and assignees.
- If auto-merge is enabled, attempt to enable or update it after the PR is updated.

When the effective result disables auto-merge:

- Leave the PR open without enabling auto-merge.
- `--no-auto-merge` only avoids enabling auto-merge during the current run. It does not disable auto-merge if the PR already has it enabled.

## Failure And Warning Semantics

The PR creation or update step should be treated as more important than the follow-up auto-merge step.

Rules:

- Invalid reviewer or assignee configuration is a precondition failure and should stop the run before PR mutation.
- Label creation failures should stop the run before PR mutation.
- Merge-method rejection must not prevent PR creation or update.
- If the desired method is rejected, retry the fallback methods in order.
- If no merge method can be enabled but the PR was created or updated successfully, keep the PR and report the result.

Recommended command outcome:

- If auto-merge came only from configuration and no method can be enabled, warn and continue after reporting the PR URL.
- If `--auto-merge` was requested explicitly and no method can be enabled, create or update the PR, report the PR URL, and then fail the command.

This preserves operator intent without sacrificing the created PR.

## Safety Rules

The CLI should only attempt auto-merge when all of the following are true:

- The run is already creating or updating a PR.
- The target repository supports auto-merge.
- The authenticated `gh` session has permission to enable auto-merge on the PR.
- The PR is not in a state that GitHub rejects immediately, such as an invalid base branch.

## Merge Method Semantics

The merge method governs the final history that lands on the base branch.

- `merge`: preserves the PR branch topology and all convention commits.
- `squash`: produces one commit on the base branch regardless of how many convention commits were created on the PR branch.
- `rebase`: replays the convention commits onto the base branch without a merge commit.

Recommended default: `squash`.

Reasoning:

- A conventions PR may contain several commits because the CLI creates one commit per convention.
- `squash` keeps the base branch history compact while still allowing the PR itself to show the per-convention commit structure during review.
- `merge` is a reasonable second choice when squash is unavailable.
- `rebase` remains acceptable, but it preserves all convention commits on the base branch and can make automation history harder to scan.

## Console Output

When a PR run computes effective behavior, print short summaries such as:

```text
Pull request labels: repo-conventions, automation
Pull request reviewers: octocat, my-org/build-team
Pull request assignees: octocat
Pull request auto-merge: enabled (desired: rebase, actual: squash, source: repository config)
```

Possible auto-merge sources:

- `cli`
- `repository config`
- `explicit convention config`
- `inferred convention defaults`
- `default`

If the desired merge method is rejected and a fallback is used, print a note explaining the fallback.

## Suggested Implementation Order

1. Add repository-level and convention-level `pull-request` settings for `labels`, `reviewers`, `assignees`, `auto-merge`, and `merge-method`.
2. Track which conventions actually produced commits and use only those conventions for convention-level pull request behavior.
3. Compute additive metadata and effective auto-merge behavior for one generated PR.
4. Query repository labels and create any missing labels, including the built-in `repo-conventions` label.
5. Apply labels, reviewers, and assignees during `gh pr create` and existing PR updates.
6. Add auto-merge enablement with desired-method selection and fallback retry behavior.
7. Add integration tests covering built-in label creation, configured label creation, commit-producing convention filtering, reviewer and assignee validation, explicit and inherited auto-merge, and merge-method fallback handling.
