# Pull Request Metadata

## Status

Proposed.

## Purpose

`repo-conventions` already creates pull requests with a stable title, a stable branch naming scheme, and a generated body that identifies the applied conventions. That means conventions PRs are already recognizable today.

This document proposes a first version of pull request metadata support for generated conventions PRs. The initial metadata surface includes:

- labels
- reviewers
- assignees

This keeps repository workflow configuration in one place under the existing `pull-request` section while leaving auto-merge behavior specified separately.

## Recommendation

Generated conventions PRs should always include the `repo-conventions` label.

That label is useful because it provides a stable GitHub-native way to scan, filter, and automate around generated PRs.

Why it helps:

- It makes conventions PRs stand out in the GitHub PR list UI.
- It enables simple searches such as `is:pr is:open label:repo-conventions`.
- It gives repositories a stable hook for automation, dashboards, and notification rules.

Why it is still not the only identifier:

- Existing conventions PRs already share the title `Apply repository conventions.`.
- Existing conventions PRs already use `repo-conventions` or `repo-conventions-N` as the head branch.
- The PR body already states that the PR was generated by `repo-conventions`.

Conclusion:

- `repo-conventions` should be an implicit label on every generated PR.
- Repositories should still be able to add more labels through configuration.
- Convention references should also be able to contribute additional labels, reviewers, and assignees.
- A future opt-out can be added later if repositories prove to need it.

## Goals

- Always add a stable `repo-conventions` label to generated conventions PRs.
- Allow repositories to attach additional labels to conventions PRs.
- Allow individual convention references to contribute labels, reviewers, and assignees.
- Make conventions PRs easier to scan in the GitHub UI.
- Support label-based automation in repositories that want it.
- Support repository workflow automation such as default reviewers and assignees.
- Keep repository owners in control of the metadata used in their repository.
- Avoid replacing the existing title and branch naming conventions as the primary PR identity.

## Non-Goals

- Inferring labels from changed files or convention names.
- Creating or managing a full repository label taxonomy.
- Removing unrelated human-added labels from an existing PR.
- Removing unrelated human-added reviewers or assignees from an existing PR.
- Making labels mandatory for `--open-pr`.
- Auto-approving pull requests.
- Re-requesting reviewers who have already completed review unless GitHub requires it.

## Background

Today, `apply --open-pr` already gives conventions PRs three stable identifiers:

- Title: `Apply repository conventions.`
- Head branch: `repo-conventions`, `repo-conventions-2`, and so on.
- Body text that explicitly says the PR was generated by `repo-conventions`.

That means labels and other metadata do not replace the current PR identity. Instead, they improve GitHub-native triage and workflow behavior around those PRs.

This matters because the built-in `repo-conventions` label should be treated differently from repository-specific metadata. The CLI can own the built-in label directly, while repository and convention configuration can add more metadata on top.

## Proposed Configuration Surface

Add metadata properties under the existing `pull-request` section in `.github/conventions.yml`.

### Repository Configuration

```yaml
pull-request:
  labels:
    - automation
  reviewers:
    - octocat
    - my-org/build-team
  assignees:
    - octocat

conventions:
  - path: owner/shared-conventions/dotnet-format@v1
```

Proposed semantics:

- `pull-request.labels` adds repository-level labels to generated conventions PRs.
- `pull-request.reviewers` adds repository-level reviewers to generated conventions PRs.
- `pull-request.assignees` adds repository-level assignees to generated conventions PRs.
- If omitted or empty, the corresponding metadata contributes nothing.
- Label matching should use GitHub's normal case-insensitive label behavior, but configuration should preserve the authored spelling.

### Convention Configuration

Allow each convention reference to contribute pull request metadata:

```yaml
conventions:
  - path: owner/shared-conventions/dotnet-format@v1
    pull-request:
      labels:
        - dependencies
      reviewers:
        - my-org/dotnet-team
      assignees:
        - octocat
```

Proposed semantics:

- Convention-level metadata is attached to the convention reference in the consuming repository.
- Convention-level metadata contributes to the generated PR as a whole.
- Convention-level metadata is additive with repository-level metadata.
- This metadata is based on the convention references that participate in the generated PR, not on metadata embedded inside remote convention repositories.

## Effective Metadata Model

One `repo-conventions` run may apply multiple conventions into one PR. For labels, reviewers, and assignees, the right reduction model is additive union rather than override.

### Labels

The effective label set for a generated PR is:

1. The implicit built-in label `repo-conventions`.
2. Repository-level `pull-request.labels`.
3. Convention-level `pull-request.labels` from applied convention references.

Rules:

- Labels are deduplicated case-insensitively.
- Existing label casing is preserved when a matching repository label already exists.
- The built-in `repo-conventions` label is always included even if it is not present in configuration.

### Reviewers

The effective reviewer set for a generated PR is the union of:

1. Repository-level `pull-request.reviewers`.
2. Convention-level `pull-request.reviewers` from applied convention references.

Rules:

- Reviewers are deduplicated.
- Reviewer values may represent either GitHub users or review-requestable teams.
- The CLI should add configured reviewers but should not remove existing reviewers.

### Assignees

The effective assignee set for a generated PR is the union of:

1. Repository-level `pull-request.assignees`.
2. Convention-level `pull-request.assignees` from applied convention references.

Rules:

- Assignees are deduplicated.
- Assignee values are GitHub users.
- The CLI should add configured assignees but should not remove existing assignees.

This model avoids the need for precedence rules because these metadata types combine naturally.

## Built-In Label

The built-in `repo-conventions` label should behave specially.

If the label does not already exist in the repository, `repo-conventions` should create it with:

- Color: `1d76db`
- Description: `Pull requests generated by repo-conventions.`

If the label already exists:

- Reuse it.
- Do not change its color.
- Do not change its description.

This gives generated PRs a predictable standard label without taking ownership of repository-local label customization.

## Label Creation

Configured labels may not exist in the repository.

Proposed behavior:

- Before creating or updating the PR, resolve the full effective label set.
- For each missing configured label, create the label automatically.
- For user-specified labels, use GitHub's default color and default empty description.
- For the built-in `repo-conventions` label, use the standard color and description defined above.
- If a label already exists, never overwrite its color or description.

Rationale:

- Repositories should not need a separate manual bootstrap step just to use PR metadata.
- Label creation is low-risk compared with reviewer or assignee assignment because labels are repository metadata, not identities.
- Preserving existing labels avoids fighting with repository-specific customization.

## Reviewer And Assignee Handling

Reviewers and assignees cannot be auto-created, so they need validation instead.

Proposed behavior:

- Validate reviewer and assignee identities before mutating the PR when GitHub exposes that information cheaply.
- If a configured reviewer or assignee is invalid or cannot be requested for the target repository, fail with a clear message.
- Do not silently skip invalid reviewers or assignees.

This keeps label setup convenient while still treating invalid identities as configuration errors.

## Behavior Model

When `apply --open-pr` creates a new PR:

- `repo-conventions` should compute the effective labels, reviewers, and assignees.
- `repo-conventions` should ensure all missing labels are created first.
- `repo-conventions` should pass the effective metadata when creating the PR.

When `apply --open-pr` updates an existing PR:

- `repo-conventions` should ensure every effective label is present.
- `repo-conventions` should ensure every effective reviewer is requested if not already present.
- `repo-conventions` should ensure every effective assignee is assigned if not already present.
- `repo-conventions` should not remove labels, reviewers, or assignees that are not part of the effective metadata set.

This makes metadata management additive and avoids fighting with human triage or other bots.

## Existing PR Utility

Would `repo-conventions` be useful when looking at existing PRs?

Yes.

Useful cases:

- Repositories with many bot-created PRs where visual scanability matters.
- Teams that use label filters, saved searches, or GitHub project workflows.
- Teams that want reporting such as "how many conventions PRs are open across repositories?"

Less useful cases:

- Repositories where conventions PRs are already obvious from the repeated title and branch naming.
- Historical PRs created before labels were introduced, unless maintainers backfill labels manually.

Recommended interpretation:

- The built-in label adds value for ongoing PR triage.
- The label is not enough reason to change the existing title or branch strategy.
- The feature should be presented as an enhancement to the current identification model, not a replacement.

## CLI Surface

The first version does not need a new CLI flag.

Reasoning:

- Pull request metadata is repository workflow policy, not typically a one-off operator choice.
- Repository config keeps repeated runs consistent.
- Avoiding CLI overrides keeps the first version simpler.

If a later need appears for one-off suppression or additional metadata, CLI overrides can be added then.

## Implementation Notes

Likely GitHub CLI integration:

- `gh label create <name>` for missing labels.
- `gh pr create --label <name>` for new PRs.
- `gh pr create --reviewer <name>` for new PRs.
- `gh pr create --assignee <name>` for new PRs.
- `gh pr edit <url> --add-label <name>` for existing PRs.
- `gh pr edit <url> --add-reviewer <name>` for existing PRs when supported, otherwise use the appropriate GitHub API operation.
- `gh pr edit <url> --add-assignee <name>` for existing PRs.

Implementation should query repository labels up front, create any missing labels before PR mutation, and validate reviewers and assignees before trying to request them.

## Suggested Implementation Order

1. Extend the planned `pull-request` configuration model with `labels`, `reviewers`, and `assignees` at both repository and convention-reference scope.
2. Compute additive effective metadata for one generated PR.
3. Query repository labels and create any missing labels, including the built-in `repo-conventions` label.
4. Apply labels, reviewers, and assignees during `gh pr create`.
5. Add missing labels, reviewers, and assignees during existing PR updates.
6. Add integration tests for built-in label creation, configured label creation, convention-level metadata aggregation, invalid reviewer or assignee handling, and preservation of unrelated PR metadata.

## Open Question

Should convention-level reviewers and assignees be contributed by every applied convention reference, or only by conventions that actually produced commits in the final PR?

Current leaning:

- Use conventions that actually contributed to the generated PR.
- That keeps metadata aligned with the changes the PR is asking humans to review.
- If later experience shows that repositories want broader metadata, repository-level configuration already covers that case.
