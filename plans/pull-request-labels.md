# Pull Request Labels

## Status

Proposed.

## Purpose

`repo-conventions` already creates pull requests with a stable title, a stable branch naming scheme, and a generated body that identifies the applied conventions. That means conventions PRs are already recognizable today.

This document proposes a small pull request label feature that improves scanning and filtering in GitHub without making labels the only way to identify conventions PRs.

## Recommendation

Yes, generated pull requests can be labeled with `repo-conventions`.

That label would be useful, but mostly as a visibility and workflow affordance rather than as a new identity mechanism.

Why it helps:

- It makes conventions PRs stand out in the GitHub PR list UI.
- It enables simple searches such as `is:pr is:open label:repo-conventions`.
- It gives repositories a stable hook for automation, dashboards, and notification rules.

Why it is not essential:

- Existing conventions PRs already share the title `Apply repository conventions.`.
- Existing conventions PRs already use `repo-conventions` or `repo-conventions-N` as the head branch.
- The PR body already states that the PR was generated by `repo-conventions`.

Conclusion:

- `repo-conventions` is a good default label name to recommend.
- The label is worth supporting because it improves filtering and scanability.
- The label should remain optional because the current title and branch naming already make old PRs identifiable.

## Goals

- Allow repositories to attach one or more stable labels to conventions PRs.
- Make conventions PRs easier to scan in the GitHub UI.
- Support label-based automation in repositories that want it.
- Keep repository owners in control of the label names used in their repository.
- Avoid replacing the existing title and branch naming conventions as the primary PR identity.

## Non-Goals

- Inferring labels from changed files or convention names.
- Supporting per-convention labels in the first version.
- Creating or managing a full repository label taxonomy.
- Removing unrelated human-added labels from an existing PR.
- Making labels mandatory for `--open-pr`.

## Background

Today, `apply --open-pr` already gives conventions PRs three stable identifiers:

- Title: `Apply repository conventions.`
- Head branch: `repo-conventions`, `repo-conventions-2`, and so on.
- Body text that explicitly says the PR was generated by `repo-conventions`.

That means a new label would not solve a discoverability gap by itself. Instead, it would make those PRs easier to recognize at a glance in list views and easier to filter with GitHub search.

This distinction matters when deciding whether `repo-conventions` should be an automatic hard-coded label or an opt-in repository policy.

## Proposed Configuration Surface

Add a `labels` property under the existing proposed `pull-request` section in `.github/conventions.yml`:

```yaml
pull-request:
  labels:
    - repo-conventions

conventions:
  - path: owner/shared-conventions/dotnet-format@v1
```

Rules:

- `pull-request.labels` is a repository-level list of labels to apply to generated conventions PRs.
- If omitted or empty, no labels are managed by `repo-conventions`.
- Label matching should use GitHub's normal case-insensitive label behavior, but configuration should preserve the authored spelling.
- The first version should support repository-level labels only.

## Why Repository-Level Only

One `repo-conventions` run may apply multiple conventions into one PR. That makes convention-level labels ambiguous in the first version.

Examples of ambiguity:

- Two conventions might request different labels.
- A repository may want one stable automation label regardless of which conventions were applied.
- A human reviewer may add labels for triage that should not be overwritten by convention-specific reduction logic.

Repository-level labels avoid those conflicts and match the way repositories usually think about PR workflow automation.

## Recommended Label Strategy

Recommended documented label name:

- `repo-conventions`

Optional companion labels that some repositories may want:

- `automation`
- `maintenance`
- `dependencies`

Recommendation for the first version:

- Document `repo-conventions` as the recommended label name.
- Do not hard-code it as an unconditional default.
- Let repositories opt in explicitly by adding it to `pull-request.labels`.

Rationale:

- Not every repository wants the same label taxonomy.
- GitHub labels are repository-local state, so hard-coding a label introduces failure cases when the label does not exist.
- Opt-in config keeps the feature predictable and makes it obvious which repositories rely on label-based workflows.

## Behavior Model

When `apply --open-pr` creates a new PR:

- `repo-conventions` should pass every configured label to `gh pr create`.

When `apply --open-pr` updates an existing PR:

- `repo-conventions` should ensure every configured label is present.
- `repo-conventions` should not remove labels that are not listed in `pull-request.labels`.

This makes label management additive and avoids fighting with human triage or other bots.

## Missing Label Handling

Configured labels may not exist in the repository.

Proposed behavior:

- Validate configured labels before creating or updating the PR.
- If any configured label does not exist, fail with a clear message naming the missing labels.
- Do not silently skip configured labels.
- Do not auto-create labels in the first version.

Rationale:

- Silent skipping would make label-based workflows unreliable.
- Auto-creating labels requires choosing colors and descriptions and would expand the scope into repository metadata management.
- A clear failure gives repository owners a one-time setup action and keeps the final PR state predictable.

## Existing PR Utility

Would `repo-conventions` be useful when looking at existing PRs?

Yes, but with limits.

Useful cases:

- Repositories with many bot-created PRs where visual scanability matters.
- Teams that use label filters, saved searches, or GitHub project workflows.
- Teams that want reporting such as "how many conventions PRs are open across repositories?"

Less useful cases:

- Repositories where conventions PRs are already obvious from the repeated title.
- Historical PRs created before labels were introduced, unless maintainers backfill labels manually.

Recommended interpretation:

- The label adds value for ongoing PR triage.
- The label is not enough reason to change the existing title or branch strategy.
- The feature should be presented as an enhancement to the current identification model, not a replacement.

## CLI Surface

The first version does not need a new CLI flag.

Reasoning:

- Labels are repository workflow policy, not typically a one-off operator choice.
- Repository config keeps repeated runs consistent.
- Avoiding CLI overrides keeps the precedence model simpler while labels remain repository-scoped.

If a later need appears for one-off labeling, repeated `--label` options can be added then.

## Implementation Notes

Likely GitHub CLI integration:

- `gh pr create --label <name>` for new PRs.
- `gh pr edit <url> --add-label <name>` for existing PRs.

Implementation should also query repository labels up front so missing labels fail early with a clear message.

## Suggested Implementation Order

1. Extend the planned `pull-request` configuration model with `labels`.
2. Validate configured labels against the target repository before PR mutation.
3. Apply labels during `gh pr create`.
4. Add missing configured labels during PR update.
5. Add integration tests for create, update, missing labels, and preservation of unrelated labels.

## Open Question

Should label validation happen only when `--open-pr` is used, or should configuration parsing fail earlier when `pull-request.labels` contains a label that the repository does not currently have?

Current leaning:

- Validate only on `--open-pr` runs.
- Parsing configuration should remain local and static.
- Repository label existence depends on remote GitHub state, so it fits better with PR-time validation than with offline config loading.
